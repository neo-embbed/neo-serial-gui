# 图表与串口高频场景性能优化记录

## 现象

程序启动后最初运行流畅，但在串口持续高速刷数据一段时间后会逐渐变卡。  
这个问题并不完全是“刷新率过高”本身导致的，更主要是缓存累计、字符串越来越大，以及高频信号造成的 UI 线程负担不断增加。

## 已确认的主要原因

### 1. 日志文本会持续增长

早期实现中，`SessionBridge` 会不断把新日志追加到 `log_`，日志内容没有有效上限。  
运行时间越久，日志字符串越大，QML 侧富文本控件处理它的成本就越高。

### 2. 日志区会重复做整段富文本重建

原先 QML 在日志变化后，会对整段日志重新执行：

- `split("\n")`
- 每行重新染色
- 重新拼接成 `RichText`

这会导致日志越长，单次刷新越重，最终出现“刚开始不卡，后来越来越卡”。

### 3. 串口消息跨 C++ / QML 边界过于频繁

原先消息链路是按“单条消息”驱动的：

- 每条串口消息都触发一次消息分发
- 每条消息都可能触发一次日志更新
- 每条消息都可能继续触发卡片解析

高频串口数据下，这种逐条触发会让主线程事件量非常大。

### 4. 卡片值更新信号过于密集

`CardBridge` 中每次卡片命中都会立刻发出 `cardValueUpdated`。  
如果同一张卡在短时间内被连续命中，就会反复唤醒 QML 更新，长期运行后会明显拖慢界面。

### 5. 卡片匹配本身是“消息数 × 卡片数”

`CardBridge.feed()` 当前仍会对每条串口数据遍历所有卡片进行匹配。  
这是当前架构下的天然复杂度来源，虽然无法完全消除，但可以通过减少上层 UI 刷新次数来降低副作用。

## 本次已实施的优化

### 1. 串口消息改为批量分发

`SessionBridge::pollMessages()` 不再按单条消息驱动界面，而是按批次向 QML 发送 `messagesReceived`。  
这样可以显著减少 C++ 与 QML 之间的调用次数。

### 2. 日志改为增量追加

QML 中的日志文本不再在每次变化时全量重建，而是仅把新到的日志行增量追加到 `richLogText`。  
只有在日志被裁剪或清空时，才会做一次整段重建。

### 3. 日志缓存增加上限

`SessionBridge` 中的日志最大行数已经限制为 `500` 行。  
超出后会自动裁剪最旧内容，避免日志无限增长。

### 4. Session 消息缓存上限缩小为 500

`core/session/session.h`

- `kMaxMessages: 20000 -> 500`

这样可以减少历史消息队列的内存占用和遍历成本。

### 5. 卡片历史缓存上限缩小为 500

`core/parameter/parameter_card.h`

- `kMaxHistory: 5000 -> 500`

这样可以减少单卡历史数据的存储与读取成本。

### 6. 卡片值更新改为“保留最新值 + 定时刷新”

`CardBridge` 中新增了待刷新队列：

- 同一张卡在一个刷新窗口内只保留最新值
- 使用 `QTimer` 每 `50ms` 批量 flush 一次
- 避免每次匹配都立刻发出一次 `cardValueUpdated`

这能明显减少高频串口数据下的 UI 信号风暴。

### 7. 日志缓存上限统一为 500

当前与“消息累计”相关的主要缓存上限已经统一收紧到 `500`：

- Session 消息缓存：`500`
- 日志行缓存：`500`
- 卡片历史缓存：`500`

## 当前效果

与最初版本相比，长时间运行后的卡顿已经明显减轻，尤其是：

- 日志区不会再因为无限增长而越来越重
- 单条消息触发的大量 UI 调用明显减少
- 高频卡片更新导致的信号风暴被削弱

## 仍然存在的剩余开销

虽然已经优化了不少，但下面这些开销仍然存在：

### 1. 每条消息仍需遍历所有卡片

`CardBridge.feed()` 当前仍是逐条消息遍历所有卡片。  
如果卡片数量很多、串口频率又很高，这部分仍会产生 CPU 压力。

### 2. QML 中 `Repeater` 下会多次调用 `CardBridge.cardAt(index)`

当前卡片 UI 的一部分数据仍是按需从 C++ 读取，而不是一次性模型化。  
这会带来额外的 C++ / QML 边界调用。

### 3. 图表绘制本身仍有一定成本

数值卡片切到 Chart 模式后，Canvas 重绘和点集遍历依然会消耗一定性能。  
不过由于历史上限已收紧，这部分的增长趋势已经被抑制。

## 后续建议

如果后面还需要继续优化，建议优先按下面顺序推进：

1. 把卡片列表从 `cardAt(index)` 方式改成一次性模型，减少 QML 对 C++ 的重复取值。
2. 继续检查 `CardBridge.feed()` 的匹配策略，评估是否可以按前缀、类型或启用状态做更早过滤。
3. 对图表重绘做更细粒度节流，只在可见且确实需要时请求重绘。
4. 如果未来日志量仍然很大，可以进一步考虑把日志显示从富文本改为更轻量的模型视图。

## 结论

这次性能问题的核心并不是单纯“刷新率太高”，而是：

- 数据累计后缓存不断变大
- 高频消息导致 UI 更新过密
- 字符串和富文本处理成本随运行时间持续增长

本轮优化已经从“缓存上限”“批量分发”“增量更新”“限频刷新”四个方向完成收敛，后续如果还要继续提升性能，建议优先处理卡片模型化和匹配路径优化。
